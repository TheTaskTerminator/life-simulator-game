## 📜 行为准则

### 我们的承诺

为了营造开放和友好的环境，我们承诺：

- 尊重所有贡献者
- 接受建设性批评
- 关注对社区最有利的事情
- 对其他社区成员表示同理心

### 不可接受的行为

- 使用性化的语言或图像
- 人身攻击、侮辱性/贬损性评论
- 公开或私下骚扰
- 未经明确许可发布他人的私人信息
- 其他在专业环境中不适当的行为

## 🚀 如何贡献

### 报告 Bug

如果你发现了 Bug，请：

1. **检查是否已存在** - 在 Issues 中搜索是否已有相同问题
2. **创建新 Issue** - 如果不存在，创建新 Issue
3. **提供详细信息**:
   - Bug 描述
   - 复现步骤
   - 预期行为
   - 实际行为
   - 环境信息（浏览器、操作系统等）
   - 截图（如适用）

### 提出功能建议

如果你有功能建议，请：

1. **检查是否已存在** - 在 Issues 中搜索是否已有相同建议
2. **创建新 Issue** - 使用 "Feature Request" 标签
3. **详细描述**:
   - 功能描述
   - 使用场景
   - 预期效果
   - 可能的实现方案（可选）

### 贡献代码

如果你想贡献代码：

1. **Fork 项目**
2. **创建特性分支** (`git checkout -b feature/AmazingFeature`)
3. **编写代码** - 遵循代码规范
4. **测试代码** - 确保代码可以正常工作
5. **提交更改** (`git commit -m 'feat: Add some AmazingFeature'`)
6. **推送到分支** (`git push origin feature/AmazingFeature`)
7. **开启 Pull Request**

## 🔄 开发流程

### 1. 设置开发环境

参考 [开发指南](./DEVELOPMENT.md) 设置开发环境。

### 2. 选择任务

- 查看 [Issues](https://github.com/your-repo/issues) 寻找可以贡献的任务
- 选择标记为 `good first issue` 的 Issue（适合新手）
- 或者提出自己的改进想法

### 3. 创建分支

```bash
# 从 main 分支创建新分支
git checkout main
git pull origin main
git checkout -b feature/your-feature-name

# 或修复 Bug
git checkout -b fix/bug-description
```

### 4. 编写代码

- 遵循 [项目规范](./STANDARDS.md)
- 编写清晰的代码
- 添加必要的注释
- 确保类型安全

### 5. 测试

- 手动测试新功能
- 确保没有破坏现有功能
- 检查控制台是否有错误

### 6. 提交代码

```bash
# 添加更改
git add .

# 提交（遵循提交规范）
git commit -m "feat: 添加新功能"

# 推送到远程
git push origin feature/your-feature-name
```

### 7. 创建 Pull Request

1. 在 GitHub 上创建 Pull Request
2. 填写 PR 描述：
   - 功能描述
   - 相关 Issue（如适用）
   - 测试说明
   - 截图（如适用）
3. 等待代码审查

## 📝 代码规范

### TypeScript

- 使用明确的类型，避免 `any`
- 遵循命名规范
- 添加必要的注释

### React

- 使用函数式组件
- 使用 Hooks 管理状态
- 使用 `React.memo` 优化性能（如需要）
- 使用 `useCallback` 和 `useMemo` 优化（如需要）

### 文件组织

- 组件放在 `components/`
- 视图放在 `views/`
- 功能 Hooks 放在 `features/`
- 服务放在 `services/`
- 工具函数放在 `utils/`

详细规范请参考 [项目规范](./STANDARDS.md)。

## 📋 提交规范

### 提交信息格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 类型 (type)

- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `style`: 代码格式（不影响代码运行）
- `refactor`: 重构（既不是新功能也不是 Bug 修复）
- `perf`: 性能优化
- `test`: 测试相关
- `chore`: 构建过程或辅助工具的变动

### 示例

```bash
# 新功能
feat(event): 添加 AI 事件生成功能

# Bug 修复
fix(attribute): 修复属性计算错误

# 文档更新
docs: 更新开发指南

# 重构
refactor(career): 重构职业系统代码结构
```

任何其他相关信息。

## ✅ Pull Request 检查清单

在提交 PR 之前，请确保：

- [ ] 代码遵循项目规范
- [ ] 代码可以正常编译
- [ ] 没有 TypeScript 错误
- [ ] 没有 ESLint 错误（如配置了）
- [ ] 已测试新功能
- [ ] 没有破坏现有功能
- [ ] 添加了必要的注释
- [ ] 更新了相关文档（如适用）
- [ ] 提交信息遵循规范

### 审查者

- 检查代码质量和规范
- 测试功能是否正常
- 提供建设性反馈

- 接受建设性批评
- 及时响应审查意见
- 根据反馈改进代码
- 保持专业和礼貌
